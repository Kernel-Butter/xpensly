# Engineering Standards & Project Architecture
## Xpensly — Field Expense Tracker

**Version:** 1.0  
**Philosophy:** Build once, reuse a hundred times. Every abstraction earns its place. Readability is not optional.

---

## 1. Core Engineering Principles

### 1.1 The Lego Rule
Every module, component, hook, and utility must be designed to snap into something else without modification. If you're copy-pasting code and changing two words, you need an abstraction. If you're writing an abstraction for one use case, you don't need it yet.

### 1.2 The Newspaper Test
Any engineer who joins this project should be able to find any piece of logic in under 30 seconds by reading the folder structure like a newspaper headline. Co-location beats convention.

### 1.3 The Zero-`any` Rule
TypeScript's value is in the types. `any` is a lie to the compiler. If you need `any`, you either don't understand the type yet (learn it) or you're at a boundary (type it with `unknown` and narrow it).

### 1.4 The Single Responsibility Rule for Files
One file = one purpose. A component file renders UI. A hook file manages one slice of state or one data concern. A utility file contains pure functions. Never mix them.

### 1.5 The Abstraction Levels Rule
Keep your code at one level of abstraction per function. A function that calls `supabase.from('expenses').select()` should not also be formatting a date string. These are different levels.

```ts

// Type vs Interface: use `type` for everything except when extending
```

## 4. Component Architecture

### 4.1 Component Categories and Rules

**`components/ui/`** — Base atoms from shadcn. Never modify these. Never import domain logic. Never know about Supabase.

**`components/shared/`** — App-specific atoms and molecules. No domain knowledge. Props-driven. Reusable across any feature. Must have no imports from `hooks/queries/` or `store/`.

```tsx
// ✅ Correct: shared component is dumb, driven by props
export function StatCard({ label, value, sub }: StatCardProps) {
  return (
    <div className="rounded-lg border border-gray-200 bg-white p-4">
      <p className="text-xs text-gray-500 uppercase tracking-wide">{label}</p>
      <p className="mt-1 font-mono text-2xl font-bold text-gray-900">{value}</p>
      {sub && <p className="mt-0.5 text-xs text-gray-400">{sub}</p>}
    </div>
  )
}

// ❌ Wrong: shared component is fetching its own data
export function StatCard({ periodId }: { periodId: string }) {
  const { data } = usePeriodSummary(periodId)  // ← This is a feature component, not shared
  ...
}
```

**`components/expense/`**, **`components/dashboard/`**, etc. — Feature components. These CAN use hooks and store. They are the composition layer between UI and data.

```tsx
// ✅ Correct: feature component orchestrates data + shared UI
export function DashboardStats() {
  const { activePeriod } = useAppStore()
  const { data: summary } = usePeriodSummary(activePeriod?.id)
  
  return (
    <div className="grid grid-cols-2 gap-3 md:grid-cols-4">
      <StatCard label="Total spent" value={formatPKR(summary?.total ?? 0)} />
      <StatCard label="Cost per acre" value={formatPKR(summary?.costPerAcre ?? 0)} />
    </div>
  )
}
```

### 4.2 The Container / Presentational Split

Every significant feature has two layers:

```
AddExpenseSheet.tsx   ← Feature component (container)
  - Owns form state (React Hook Form)
  - Calls useAddExpense mutation
  - Reads business config from store
  - Passes data down to pure UI

AddExpenseForm.tsx    ← Presentational component
  - Receives: categories, workers, onSubmit, isSubmitting
  - Returns: JSX only
  - Zero side effects
  - Fully testable without mocking anything
```

### 4.3 Composition over Props Drilling

Never pass props more than 2 levels deep. Use one of:

1. **React Context** for feature-scoped state (e.g., the add-expense sheet's form state)
2. **Zustand** for global app state
3. **Component composition** — pass children or render props instead

```tsx
// ❌ Props drilling anti-pattern
<ExpenseList
  businessId={businessId}
  periodId={periodId}
  currency={currency}
  categories={categories}
  onExpenseDelete={handleDelete}
  onExpenseEdit={handleEdit}
/>

// ✅ Hook pulls what it needs; component stays clean
export function ExpenseList() {
  const { activePeriod } = useAppStore()
  const { data: expenses } = useExpenses(activePeriod?.id)
  
  return expenses?.map(exp => <ExpenseRow key={exp.id} expense={exp} />)
}
```

## 5. Hooks Architecture

### 5.1 Query Hooks (TanStack Query)

Each query hook wraps one Supabase query. One file per query. No logic besides the query.

```ts
// hooks/queries/useExpenses.ts
import { useQuery } from '@tanstack/react-query'
import { supabase } from '@/lib/supabase/client'
import type { Expense } from '@/types'

export const expenseKeys = {
  all:    () => ['expenses'] as const,
  byPeriod: (periodId: string) => ['expenses', 'period', periodId] as const,
}

export function useExpenses(periodId: string | undefined) {
  return useQuery({
    queryKey: expenseKeys.byPeriod(periodId ?? ''),
    queryFn: async () => {
      if (!periodId) return []
      
      const { data, error } = await supabase
        .from('expenses')
        .select('*, worker:workers(id, name)')
        .eq('period_id', periodId)
        .order('date', { ascending: false })
        .order('created_at', { ascending: false })
      
      if (error) throw error
      return data as Expense[]
    },
    enabled: !!periodId,
    staleTime: 1000 * 60 * 2,  // 2 minutes
  })
}
```

### 5.2 Mutation Hooks (TanStack Query)

Each mutation hook handles one operation: add, edit, or delete. Includes optimistic update.

```ts
// hooks/mutations/useAddExpense.ts
import { useMutation, useQueryClient } from '@tanstack/react-query'
import { supabase } from '@/lib/supabase/client'
import { expenseKeys } from '@/hooks/queries/useExpenses'
import { offlineQueue } from '@/lib/offline/db'
import type { ExpenseInput, Expense } from '@/types'

export function useAddExpense(periodId: string) {
  const queryClient = useQueryClient()
  
  return useMutation({
    mutationFn: async (input: ExpenseInput): Promise<Expense> => {
      // If offline, queue it and return a temp record
      if (!navigator.onLine) {
        await offlineQueue.add({ ...input, period_id: periodId })
        return { ...input, id: `temp-${Date.now()}`, period_id: periodId } as Expense
      }
      
      const { data, error } = await supabase
        .from('expenses')
        .insert({ ...input, period_id: periodId })
        .select()
        .single()
      
      if (error) throw error
      return data as Expense
    },
    
    // Optimistic update: show the new expense instantly
    onMutate: async (newExpense) => {
      await queryClient.cancelQueries({ queryKey: expenseKeys.byPeriod(periodId) })
      
      const previous = queryClient.getQueryData<Expense[]>(expenseKeys.byPeriod(periodId))
      
      queryClient.setQueryData<Expense[]>(expenseKeys.byPeriod(periodId), (old = []) => [
        { ...newExpense, id: `temp-${Date.now()}`, period_id: periodId } as Expense,
        ...old,
      ])
      
      return { previous }  // context for onError rollback
    },
    
    // On error: revert to previous state
    onError: (_err, _vars, context) => {
      if (context?.previous) {
        queryClient.setQueryData(expenseKeys.byPeriod(periodId), context.previous)
      }
    },
    
    // On success: replace temp record with real one from server
    onSettled: () => {
      queryClient.invalidateQueries({ queryKey: expenseKeys.byPeriod(periodId) })
      queryClient.invalidateQueries({ queryKey: ['period-summary', periodId] })
    },
  })
}
```

### 5.3 Zustand Store

One store file. Slices by concern within the file. Use `immer` middleware for nested updates.

```ts
// store/app.store.ts
import { create } from 'zustand'
import { persist } from 'zustand/middleware'
import { immer } from 'zustand/middleware/immer'
import type { Business, Context, Period, Expense } from '@/types'

type AppState = {
  // Persisted selections
  activeBusiness:  Business | null
  activeContext:   Context | null
  activePeriod:    Period | null
  
  // UI-only state
  isAddExpenseOpen: boolean
  editingExpense:   Expense | null
}

type AppActions = {
  setActiveBusiness:  (business: Business) => void
  setActiveContext:   (context: Context) => void
  setActivePeriod:    (period: Period) => void
  openAddExpense:     () => void
  closeAddExpense:    () => void
  openEditExpense:    (expense: Expense) => void
  closeEditExpense:   () => void
}

export const useAppStore = create<AppState & AppActions>()(
  persist(
    immer((set) => ({
      // Initial state
      activeBusiness:   null,
      activeContext:    null,
      activePeriod:     null,
      isAddExpenseOpen: false,
      editingExpense:   null,
      
      // Actions
      setActiveBusiness:  (b) => set((s) => { s.activeBusiness = b }),
      setActiveContext:   (c) => set((s) => { s.activeContext = c }),
      setActivePeriod:    (p) => set((s) => { s.activePeriod = p }),
      openAddExpense:     () => set((s) => { s.isAddExpenseOpen = true }),
      closeAddExpense:    () => set((s) => { s.isAddExpenseOpen = false }),
      openEditExpense:    (e) => set((s) => { s.editingExpense = e; s.isAddExpenseOpen = true }),
      closeEditExpense:   () => set((s) => { s.editingExpense = null; s.isAddExpenseOpen = false }),
    })),
    {
      name: 'xpensly-app',
      partialize: (state) => ({          // Only persist selections, not UI state
        activeBusiness: state.activeBusiness,
        activeContext:  state.activeContext,
        activePeriod:   state.activePeriod,
      }),
    }
  )
)
```

## 7. Utility Functions

All utilities are pure functions. No side effects. No imports from React or Supabase.

```ts
// lib/utils/currency.ts
export function formatCurrency(amount: number, currency = 'PKR'): string {
  return new Intl.NumberFormat('en-PK', {
    style: 'currency',
    currency,
    maximumFractionDigits: 0,
  }).format(amount)
}

export function abbreviateCurrency(amount: number, currency = 'PKR'): string {
  if (amount >= 100_000) return `${(amount / 100_000).toFixed(2)}L`
  if (amount >= 1_000)   return `${(amount / 1_000).toFixed(1)}K`
  return formatCurrency(amount, currency)
}

// lib/utils/math.ts
export function calculateTotal(quantity: number | null, unitCost: number | null): number | null {
  if (quantity === null || unitCost === null) return null
  return quantity * unitCost
}

export function costPerUnit(totalCost: number, unitSize: number): number {
  if (unitSize === 0) return 0
  return totalCost / unitSize
}

// lib/utils/date.ts
export function formatDate(date: string | Date, format: 'short' | 'long' | 'relative' = 'short'): string {
  const d = new Date(date)
  if (format === 'relative') return formatRelative(d)
  if (format === 'long') return d.toLocaleDateString('en-PK', { day: 'numeric', month: 'long', year: 'numeric' })
  return d.toLocaleDateString('en-PK', { day: 'numeric', month: 'short' })
}

export function groupByDate<T extends { date: string }>(items: T[]): Record<string, T[]> {
  return items.reduce<Record<string, T[]>>((acc, item) => {
    const key = item.date
    return { ...acc, [key]: [...(acc[key] ?? []), item] }
  }, {})
}

// lib/utils/cn.ts — Used everywhere for conditional classes
import { clsx, type ClassValue } from 'clsx'
import { twMerge } from 'tailwind-merge'

export function cn(...inputs: ClassValue[]): string {
  return twMerge(clsx(inputs))
}
```

## 8. Server Components vs Client Components

**Decision rule:** Start with a Server Component. Add `'use client'` only when you need:
- `useState` or `useReducer`
- `useEffect`
- Browser APIs (`window`, `navigator`, `localStorage`)
- Event listeners
- Third-party client libraries (Recharts, etc.)

```tsx
// ✅ Server component — fetches data, no interactivity
// app/(app)/page.tsx
import { createSupabaseServerClient } from '@/lib/supabase/server'
import { DashboardStats } from '@/components/dashboard/DashboardStats'

export default async function DashboardPage() {
  const supabase = createSupabaseServerClient()
  const { data: businesses } = await supabase.from('businesses').select('*')
  
  return <DashboardStats initialData={businesses} />
}

// ✅ Client component — needs interactivity
// components/expense/AddExpenseSheet.tsx
```

## 9. Error Handling

### 9.1 Layer-by-Layer Error Strategy

```
Supabase errors     → caught in query/mutation hooks, thrown as JS Error
Mutation errors     → TanStack Query retries + onError rollback + toast
Form errors         → Zod + React Hook Form inline field errors
Route errors        → error.tsx boundary catches unhandled errors
Network offline     → detected pre-request, routed to offline queue
Unknown runtime     → Sentry (optional) + error.tsx fallback
```

### 9.2 Custom Error Types

```ts
// types/errors.types.ts
export class DatabaseError extends Error {
  constructor(
    message: string,
    public readonly code: string,
    public readonly details?: string
  ) {
    super(message)
    this.name = 'DatabaseError'
  }
}

export class AuthError extends Error {
  constructor(message: string) {
    super(message)
    this.name = 'AuthError'
  }
}

export class ValidationError extends Error {
  constructor(
    message: string,
    public readonly fields: Record<string, string[]>
  ) {
    super(message)
    this.name = 'ValidationError'
  }
}
```

```ts
// Validates shape at compile time without widening the type
```

```ts
type SyncState =
  | { status: 'idle' }

// Now TypeScript knows which fields are available per state:
function SyncIndicator({ state }: { state: SyncState }) {
  if (state.status === 'error') {
    return <span>{state.message}</span>  // TypeScript knows message exists
  }
  // ...
}
```

### 10.3 Generic Hooks

```ts
// Reusable paginated query pattern
function usePaginatedQuery<TData>(
  queryKey: readonly unknown[],
  fetcher: (cursor: string | null) => Promise<{ data: TData[]; nextCursor: string | null }>
) {
  return useInfiniteQuery({
    queryKey,
    queryFn: ({ pageParam }) => fetcher(pageParam as string | null),
    getNextPageParam: (last) => last.nextCursor,
    initialPageParam: null,
  })
}

// Usage (no boilerplate):
```

## 13. Performance Patterns

### 13.1 Memoization — Use Sparingly

```tsx
// ❌ Don't memo everything — it has a cost
const MemoRow = memo(ExpenseRow)  // Only if profiling shows re-render problem

// ✅ useMemo only for expensive derivations
const categoryTotals = useMemo(
  () => aggregateByCategory(expenses),
  [expenses]  // Only re-runs when expenses changes
)

// ✅ useCallback only for stable function references passed to children
const handleDelete = useCallback(
  (id: ExpenseId) => deleteExpense(id),
  [deleteExpense]
)
```

### 13.2 Code Splitting

```tsx
// Heavy libraries: lazy-load at route level
const ReportsPage = dynamic(() => import('@/components/reports/ReportsPage'), {
```

### 13.3 List Virtualisation (if list exceeds 200 rows)

```tsx
// Use @tanstack/react-virtual for long expense lists
import { useVirtualizer } from '@tanstack/react-virtual'

export function ExpenseList({ expenses }: { expenses: Expense[] }) {
  const parentRef = useRef<HTMLDivElement>(null)
  const virtualizer = useVirtualizer({
    count: expenses.length,
    getScrollElement: () => parentRef.current,
    estimateSize: () => 64,  // Row height estimate
  })
  
  return (
    <div ref={parentRef} style={{ height: '100%', overflow: 'auto' }}>
      <div style={{ height: virtualizer.getTotalSize() }}>
        {virtualizer.getVirtualItems().map((item) => (
          <ExpenseRow key={expenses[item.index].id} expense={expenses[item.index]} />
        ))}
      </div>
    </div>
  )
}
```
